## Summary

DIVD has started a new research effort into unintended data exposure in applications built on the Salesforce Experience Cloud platform, specifically related to the Aura framework.

The issue involves misconfigured access controls in Experience Cloud environments that may allow unauthenticated (guest) users or low-privileged users to access backend data through publicly exposed Aura endpoints.

It is important to note that this issue does not represent a vulnerability in the Salesforce platform itself. Salesforce provides robust security mechanisms, including object-level permissions, field-level security, sharing rules, and guest user access restrictions. However, when these controls are not correctly configured, unauthorized access to sensitive data may occur.

Typical causes include:

- overly permissive Guest User profile permissions
- misconfigured object- and field-level security (OLS/FLS)
- exposed Aura endpoints (/s/sfsites/aura or /aura) accessible without proper restrictions
- missing or overly broad sharing rules
- excessive access granted to Experience Cloud guest users
- improper configuration of Apex controllers used by Aura components
- lack of enforcement of CRUD/FLS checks in custom Apex logic

This type of exposure has been increasingly observed in the wild and is actively being exploited.

Recent reporting and threat intelligence indicate that threat actors, including groups such as ShinyHunters, claim to be actively harvesting data from exposed Salesforce Aura endpoints at scale.

Due to the recurring nature of these misconfigurations and the growing number of exposed applications, DIVD has initiated this research effort to assess the exposure of Salesforce Experience Cloud environments on the internet.

In some cases, the research includes interacting with publicly accessible endpoints to verify whether access controls are properly enforced.

## Affected scope

This issue is not limited to a specific Salesforce version or product. It may affect:

- Salesforce Experience Cloud (Community / Portal) deployments
- Public-facing Salesforce portals using Aura components
- Custom implementations using Aura or Lightning components
- Environments where guest user access is enabled

Because the issue originates from configuration and implementation choices, it may occur in any Salesforce environment where access controls are not properly configured.

## How attackers can misuse this

No exploit is required to retrieve exposed data.

Salesforce Experience Cloud applications use the Aura framework to communicate between client and server. Requests are typically sent to endpoints such as:

```
/s/sfsites/aura
/aura
```

These endpoints accept structured requests that invoke server-side controllers and return serialized data.

Because the client-side application runs in the browser, it must always be considered untrusted.

If access controls are not properly enforced, an attacker can:

- enumerate accessible Aura actions
- invoke backend controllers directly
- retrieve object data via standard or custom Apex controllers
- automate large-scale data extraction across multiple environments

This makes the issue:

- difficult to detect through traditional monitoring
- easy to automate using scripts or tooling
- highly scalable across many exposed portals

Observed exposed data in real-world cases includes:

- names, email addresses, and phone numbers
- customer and account records
- support tickets and case data
- internal identifiers and metadata
- documents and attachments

Such exposure may lead to:

- privacy violations
- phishing and social engineering attacks
- fraud and account takeover attempts
- reputational damage
- regulatory consequences (e.g., GDPR/AVG breach notifications)

## Root cause

Salesforce Experience Cloud applications rely on a combination of frontend components (Aura/Lightning) and backend logic (Apex controllers and Salesforce data model).

Data exposure typically occurs when access controls are not properly enforced across these layers.

Common root causes include:

- overly permissive Guest User profile permissions
- misconfigured object-level security (CRUD)
- missing or incorrect field-level security (FLS) enforcement
- overly broad sharing rules or public data visibility
- Apex controllers that do not enforce security checks (with sharing, stripInaccessible, etc.)
- Aura-enabled methods (@AuraEnabled) exposing sensitive data without validation
- improper restriction of accessible Aura endpoints

These issues are related to application configuration and development practices, not to a vulnerability in the Salesforce platform itself.

## What you can do

Organizations using Salesforce Experience Cloud are strongly advised to review their security configuration.

Recommended checks include:

- Review Guest User profile permissions
- Restrict object-level (CRUD) access to only required objects
- Enforce field-level security (FLS) for all exposed data
- Review and limit sharing rules and data visibility
- Ensure Apex controllers enforce security (with sharing, stripInaccessible)
- Review all @AuraEnabled methods for proper authorization checks
- Disable unnecessary guest user access
- Restrict access to Aura endpoints where possible
- Implement logging and monitoring for unusual data access patterns
- Perform a security review or penetration test focused on Experience Cloud exposure

If sensitive data is accessible:

- restrict access immediately
- rotate credentials and review access tokens if applicable
- review logs for signs of abuse or large-scale data extraction
- assess whether a data breach notification is required under applicable regulations

## What we are doing

DIVD is analysing publicly accessible Salesforce Experience Cloud applications to identify instances where authorization misconfiguration may expose data.

The Salesforce Experience Cloud instances included in this research were identified through publicly available sources. Because of this, the dataset used during this research is not exhaustive and may not include all Salesforce Experience Cloud applications.

Organizations whose applications appear to expose data that is likely not intended to be publicly accessible may be notified with information about the observed behaviour. Organizations are encouraged to proactively review their Salesforce Experience Cloud applications for potential authorization misconfigurations.

Organizations that would like additional information about this research effort can contact the DIVD CSIRT team at DIVD-2026-00004@csirt.divd.nl.
